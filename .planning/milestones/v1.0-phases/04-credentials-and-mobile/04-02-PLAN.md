<objective>
Implement organization delegate routing: when an approval targets an org entity, fan-out the DIDComm message to all delegates (up to 50), enforce first-response-wins semantics, and apply per-source rate limiting to prevent fan-out abuse.

Purpose: Organizations need their members to respond to approval requests. MSG-06 enables this by routing to delegates with race-safe first-response-wins.
Output: `store/org_delegates.go` query, updated `CreateApproval` handler with fan-out logic and per-source rate limiting, tests.
</objective>

  <action>
    1. Add `OrgDelegateStore` interface to `api.go`:
       ```go
       type OrgDelegateStore interface {
           GetOrgDelegates(ctx context.Context, orgDID string, limit int) ([]models.Entity, error)
       }
       ```
       Note: *Store already implements this since we added it in Task 1. Add the interface field to Handler and update NewHandler signature. The Store satisfies both EntityStore and OrgDelegateStore.

    2. In `approvals.go` CreateApproval handler, after creating the approval and dispatching DIDComm to `to`:
       - Look up the `to` entity. If entity.Type == "org", call `GetOrgDelegates(ctx, entity.DID, 50)`.
       - For each delegate, queue a DIDComm message with the approval request, setting `To` to the delegate's DID.
       - Use goroutines for concurrent dispatch (same pattern as research Pattern 4).

    3. **Per-source rate limiting for org fan-out (MSG-06):** Before executing fan-out, check a Redis counter to enforce per-source rate limits. Use the same pattern as OTP rate limiting in Plan 03:
       - Redis key: `fanout:rate:{source_did}:{org_did}` with 1-hour TTL
       - On each fan-out request, INCR the counter. If counter > 10, return 429 Too Many Requests with RFC 7807 problem detail type `https://atap.dev/errors/rate-limit-exceeded`, title "Rate limit exceeded", detail "Too many fan-out requests to this organization. Try again later."
       - The 10-per-hour-per-org threshold prevents a single requester from flooding an org's delegates. The limit is per (source_did, org_did) pair so different requesters do not interfere.
       - Use `INCR` + `EXPIRE` (set TTL only if key is new via `NX` flag on the EXPIRE, or use Redis MULTI/EXEC to atomically INCR and conditionally set TTL).

    4. First-response-wins is already handled by the existing `UpdateApprovalState` which uses `WHERE state='requested'` — if RowsAffected == 0, the approval was already handled. Verify this is the case in RespondApproval. If RespondApproval does not currently check rows affected, add the check and return a 409 Conflict (RFC 7807 `already-responded` error) instead of 500.

    5. Update existing approval tests to cover org fan-out scenarios. Add mock for OrgDelegateStore in test helpers. Add rate limit test: mock Redis to return counter values above and below threshold.
