# Phase 3: Mobile App - Research

**Researched:** 2026-03-11
**Domain:** Flutter mobile app + Go platform push notification extensions
**Confidence:** MEDIUM

## Summary

This phase spans two codebases: a new Flutter mobile app (greenfield) and extensions to the existing Go platform. The Flutter app must generate Ed25519 keypairs, store private keys securely with biometric protection, connect to the platform via REST and SSE, and handle push notifications via FCM/APNs. The Go platform needs new endpoints (claims, human registration, delegations, push tokens), new database tables, and a push notification delivery system integrated into the existing signal pipeline.

The highest-risk item is **Ed25519 cross-language compatibility** between Dart and Go -- the CONTEXT.md correctly identifies this as the first task to validate. The `ed25519_edwards` Dart package is a direct port of Go's `crypto/ed25519` and is the safest choice for byte-level compatibility. Push notification delivery via the official Firebase Admin Go SDK (`firebase.google.com/go/v4`) is the standard approach, handling both FCM (Android) and APNs (iOS) through a single API.

**Primary recommendation:** Use `ed25519_edwards` for Dart/Go crypto compatibility, `flutter_secure_storage` with biometric options for key storage, Riverpod for state management, `firebase_messaging` for client-side push, and the official Firebase Admin Go SDK for server-side push delivery.

### Pattern 1: Ed25519 Signed Request Auth from Flutter
**What:** Flutter app signs every API request with the user's Ed25519 private key, matching the existing Go auth middleware.
**When to use:** Every authenticated API call from the mobile app.
**Example:**
```dart
// Dart: Sign a request using ed25519_edwards (matches Go crypto.SignRequest)
import 'package:ed25519_edwards/ed25519_edwards.dart' as ed;
import 'dart:convert';

String signRequest(ed.PrivateKey privateKey, String keyId, String method, String path, DateTime ts) {
  final timestamp = ts.toUtc().toIso8601String();  // RFC3339
  final payload = '$method $path $timestamp';
  final signature = ed.sign(privateKey, utf8.encode(payload));
  final sigB64 = base64Url.encode(signature).replaceAll('=', '');
  return 'Signature keyId="$keyId",algorithm="ed25519",headers="(request-target) x-atap-timestamp",signature="$sigB64"';
}
```

### Pattern 2: Push Notification Delivery (Platform-Side)
**What:** When a signal is saved and published to Redis, also send a push notification via FCM/APNs.
**When to use:** Every signal delivery to an entity that has a push token registered.
**Example:**
```go
// Go: Integrate push delivery into existing SendSignal flow
// After Redis publish in api.go SendSignal handler:
if h.pushService != nil {
    go h.pushService.SendNotification(ctx, targetID, sig)
}
```

### Pattern 3: SSE + Push Dual Delivery
**What:** SSE for foreground real-time updates; push notifications for background/terminated state.
**When to use:** Inbox screen listens to SSE while app is active. Push fires regardless.

### Anti-Patterns to Avoid
- **Polling instead of SSE:** The platform already supports SSE streams. Do not build a timer-based polling mechanism.
- **Storing private keys in SharedPreferences:** MUST use flutter_secure_storage with biometric protection. SharedPreferences is plaintext on disk.
- **Client-side push filtering:** Per locked decision, all signals trigger push. Do not add client-side filtering logic.
- **Generating keys on the server:** Human entities generate their Ed25519 keypair ON DEVICE. The server never sees the private key.
- **Hardcoding Firebase credentials in code:** Use `google-services.json` (Android) and `GoogleService-Info.plist` (iOS) from Firebase console.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Ed25519 signing | Custom Dart FFI to libsodium | `ed25519_edwards` package | Direct Go port, pure Dart, no native compilation issues |
| Secure key storage | Custom encrypted file storage | `flutter_secure_storage` | Wraps iOS Keychain + Android Keystore with biometric options |
| Push notification delivery | Raw HTTP to FCM/APNs endpoints | Firebase Admin Go SDK | Handles auth, token refresh, APNs certificate management, batch sending |
| SSE parsing | Manual HTTP stream parsing | `flutter_client_sse` or `flutter_http_sse` | Handles reconnection, event parsing, Last-Event-ID |
| Deep link handling | Manual intent filter parsing | `app_links` | Handles both universal links (iOS) and app links (Android) |
| JSON serialization | Manual fromJson/toJson | `freezed` + `json_serializable` | Type-safe, immutable, eliminates boilerplate errors |
| Human ID derivation | Custom base32 implementation | Use `ed25519_edwards` key bytes + Dart crypto SHA-256 + Dart base32 | Must match Go's `crypto.DeriveHumanID` byte-for-byte |

**Key insight:** The critical risk in this phase is cross-language crypto compatibility. Using `ed25519_edwards` (which IS Go's ed25519 ported to Dart) eliminates the biggest class of integration bugs.

## Common Pitfalls

### Pitfall 1: Ed25519 Key Format Mismatch
**What goes wrong:** Dart and Go produce different byte representations of the same key, causing signature verification failures.
**Why it happens:** Different Ed25519 libraries use different internal key formats (64-byte vs 32-byte seeds, different encoding of public keys).
**How to avoid:** Use `ed25519_edwards` which is a literal port of Go's `crypto/ed25519`. Validate in the FIRST task: generate keypair in Dart, sign payload, verify in Go.
**Warning signs:** "Invalid signature" errors on the first API call from the mobile app.

### Pitfall 2: Base64 Encoding Variants
**What goes wrong:** Dart's `base64Url.encode()` adds padding (`=`), Go's `base64.RawURLEncoding` does not.
**Why it happens:** RFC 4648 allows both padded and unpadded base64url.
**How to avoid:** Always strip padding in Dart: `.replaceAll('=', '')`. The existing Go code uses `base64.RawURLEncoding` (no padding).
**Warning signs:** Auth header parsing failures with trailing `=` characters.

### Pitfall 3: SSE Connection Management on Mobile
**What goes wrong:** SSE connections are held open, draining battery and data when app is backgrounded.
**Why it happens:** Mobile apps should disconnect SSE when entering background state.
**How to avoid:** Listen to `AppLifecycleState` changes. Disconnect SSE on `paused`/`inactive`, reconnect with Last-Event-ID on `resumed`.
**Warning signs:** Battery drain complaints, data usage spikes, orphaned server connections.

### Pitfall 4: FCM Token Refresh
**What goes wrong:** Push notifications stop working after token refresh.
**Why it happens:** FCM tokens are periodically rotated by Google. If the platform stores a stale token, pushes fail silently.
**How to avoid:** Listen to `FirebaseMessaging.instance.onTokenRefresh` and POST updated token to platform. Platform should upsert (not insert) push tokens.
**Warning signs:** Push notifications work initially but stop after days/weeks.

### Pitfall 5: APNs Token Not Ready on iOS
**What goes wrong:** FCM plugin API calls fail on iOS because APNs token hasn't arrived yet.
**Why it happens:** iOS requires APNs token before FCM can register. This is async and not guaranteed before app init completes.
**How to avoid:** Call `getAPNSToken()` before `getToken()` on iOS. Wait for APNs token availability.
**Warning signs:** Token is null on first launch on iOS, but works on Android.

### Pitfall 6: Claim Code Deep Link Race Condition
**What goes wrong:** App receives deep link but is not yet initialized (cold start).
**Why it happens:** Universal/app links can launch the app from a terminated state.
**How to avoid:** `app_links` provides both `getInitialLink()` (cold start) and `uriLinkStream` (warm start). Handle BOTH.
**Warning signs:** Deep links work when app is already running but fail from cold start.

### Pitfall 7: Biometric Auth on Emulator
**What goes wrong:** Tests fail on emulators without biometric hardware.
**Why it happens:** `flutter_secure_storage` biometric options require real hardware.
**How to avoid:** Make biometric requirement configurable for debug builds. Use `canAuthenticate()` check before requiring biometrics.
**Warning signs:** App crashes or hangs on emulator during key access.

## Code Examples

### Human Registration Request (Dart -> Go)
```dart
// Dart: Generate keypair, derive human ID, register with platform
import 'package:ed25519_edwards/ed25519_edwards.dart' as ed;
import 'dart:convert';
import 'package:crypto/crypto.dart'; // for sha256

Future<void> registerHuman(String email, String claimCode) async {
  // Generate keypair on device
  final keyPair = ed.generateKey();
  final publicKeyBytes = keyPair.publicKey.bytes;

  // Derive human ID (must match Go: lowercase(base32(sha256(pubkey))[:16]))
  final hash = sha256.convert(publicKeyBytes);
  final humanId = base32Encode(hash.bytes).substring(0, 16).toLowerCase();

  // Store private key with biometric protection
  await secureStorage.write(
    key: 'private_key',
    value: base64.encode(keyPair.privateKey.bytes),
    aOptions: AndroidOptions(
      encryptedSharedPreferences: true,
      resetOnError: true,
    ),
    iOptions: IOSOptions(
      accessibility: KeychainAccessibility.first_unlock_this_device,
    ),
  );

  // Register with platform
  final body = jsonEncode({
    'public_key': base64.encode(publicKeyBytes),
    'email': email,
    'claim_code': claimCode,
  });
  // ... sign and send request
}
```

### Push Token Registration (Go Platform)
```go
// Go: POST /v1/entities/:entityId/push-token
type PushTokenRequest struct {
    Token    string `json:"token"`    // FCM registration token
    Platform string `json:"platform"` // "android" or "ios"
}

// Store: upsert to handle token refresh
func (s *Store) UpsertPushToken(ctx context.Context, entityID, token, platform string) error {
    _, err := s.pool.Exec(ctx, `
        INSERT INTO push_tokens (entity_id, token, platform, created_at, updated_at)
        VALUES ($1, $2, $3, NOW(), NOW())
        ON CONFLICT (entity_id) DO UPDATE
        SET token = $2, platform = $3, updated_at = NOW()`,
        entityID, token, platform)
    return err
}
```

### Push Notification Delivery (Go Platform)
```go
// Go: Send push notification when signal is saved
import (
    firebase "firebase.google.com/go/v4"
    "firebase.google.com/go/v4/messaging"
)

func (p *PushService) SendNotification(ctx context.Context, entityID string, sig *models.Signal) {
    token, err := p.store.GetPushToken(ctx, entityID)
    if err != nil || token == nil {
        return // no push token registered
    }

    message := &messaging.Message{
        Token: token.Token,
        Notification: &messaging.Notification{
            Title: sig.Signal.Type,
            Body:  fmt.Sprintf("from %s", sig.Route.Origin),
        },
        Data: map[string]string{
            "signal_id": sig.ID,
            "type":      sig.Signal.Type,
        },
    }

    _, err = p.fcmClient.Send(ctx, message)
    if err != nil {
        p.log.Error().Err(err).Str("entity_id", entityID).Msg("push send failed")
    }
}
```

## Open Questions

1. **Email verification mechanism**
   - What we know: Email verification is required for Trust Level 1. Platform needs to send a verification email.
   - What's unclear: Use a third-party email service (SendGrid, SES) or simple SMTP? What does the verification flow look like (link vs code)?
   - Recommendation: Use a 6-digit code sent via SMTP for simplicity. Platform generates code, stores hash, verifies on submission. Defer full email service integration to production deployment.

2. **Firebase project setup**
   - What we know: Need Firebase project with FCM enabled for both Android and iOS.
   - What's unclear: Whether the project owner has a Firebase project created, Apple Developer account for APNs keys.
   - Recommendation: Document setup steps. Use `flutterfire configure` CLI tool to generate configs. APNs key upload to Firebase console is a manual prerequisite.

3. **Claim link landing page hosting**
   - What we know: `https://link.atap.app/claim/ATAP-X7K2` needs to serve a fallback page for users without the app.
   - What's unclear: Hosting infrastructure for the landing page.
   - Recommendation: Simple static HTML page hosted on the same platform domain or a CDN. Can be a single HTML file served by the Go platform itself at the claim URL path.

4. **Human entity ID storage in entities table**
   - What we know: Human IDs are derived from public key (not ULID). Existing entities table uses ULID for ID.
   - What's unclear: Whether to store derived human ID in the same `id` column or add a `human_id` column.
   - Recommendation: Store derived human ID in the `id` column. The entity type field distinguishes humans from agents. URI becomes `human://{derived_id}`.
