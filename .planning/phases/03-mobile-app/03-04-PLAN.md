  <action>
    1. Create `sse_client.dart`:
       - SseClient class with baseUrl, auth headers
       - `connect(String entityId, {String? lastEventId})` -> Stream of Signal objects
       - Uses http package to open GET /v1/inbox/{entityId}/stream with signed request headers
       - Parses SSE format: "data: {...}\n\n" events, extracts JSON, deserializes to Signal
       - Handles "id:" field for Last-Event-ID tracking
       - Ignores comment lines (heartbeat ": heartbeat\n")
       - `disconnect()` method to close the HTTP connection
       - Auto-stores last event ID for reconnection
       - Listen to WidgetsBindingObserver AppLifecycleState:
         - On `paused`/`inactive`: disconnect SSE (battery saving per pitfall 3)
         - On `resumed`: reconnect with stored Last-Event-ID

    2. Create `claim_screen.dart`:
       - Receives claim code from route parameter (/claim/:code)
       - Also handles cold-start deep links via getInitialLink() (per pitfall 6)
       - Shows claim code prominently, loading indicator while verifying
       - Calls GET /v1/claims/:code to verify claim is valid and pending
       - If valid: shows "You've been invited by {creator agent}" and "Get Started" button
       - If invalid/expired: shows error message with explanation
       - "Get Started" button navigates to register_screen with claim code

    3. Create `register_screen.dart`:
       - Receives claim code from navigation arguments
       - Email input field with basic validation (contains @, not empty)
       - "Register" button triggers:
         a. Generate Ed25519 keypair via ed25519_service.generateKeyPair()
         b. Store private key in secure storage with biometric protection
         c. Call apiClient.registerHuman(base64(publicKey), email, claimCode)
         d. On success: store entity ID and key ID in secure storage, update auth provider
         e. Navigate to /inbox
       - Loading state during registration
       - Error handling: show RFC 7807 error messages (claim already redeemed, invalid code, etc.)
       - Biometric prompt appears when storing the private key (per locked decision: biometric required for every signed action)
  </action>

    1. Create `inbox_provider.dart`:
       - Riverpod StateNotifier or AsyncNotifier managing List of Signal
       - `loadInbox()` -- calls GET /v1/inbox/{entityId}?limit=50, replaces signal list
       - `loadMore(String cursor)` -- pagination, appends to list
       - `connectSSE()` -- opens SSE stream via SseClient, prepends new signals to list
       - `disconnectSSE()` -- closes SSE stream
       - `refresh()` -- disconnects SSE, reloads from API, reconnects SSE
       - Exposes: signals (List<Signal>), isLoading, hasMore, isStreaming

    2. Create `inbox_screen.dart` -- card-based list layout (per locked decision):
       - Riverpod Consumer widget watching inbox provider
       - RefreshIndicator wrapping ListView.builder for pull-to-refresh
       - Each signal renders as a Material Card showing:
         - Sender (route.origin) -- entity URI or "Unknown"
         - Signal type (signal.type) -- e.g., "task.completed"
         - Timestamp (created_at) -- relative time ("2m ago") or date
         - Data preview -- first 100 chars of JSON.encode(signal.data), truncated
         - Priority indicator -- colored dot or badge based on context.priority (high=red, normal=blue, low=grey)
       - Tap on card navigates to /inbox/:signalId (signal detail)
       - Empty state: centered "No signals yet" text with pull-to-refresh hint
       - On screen init: loadInbox() then connectSSE()
       - On dispose: disconnectSSE()
       - Infinite scroll: when reaching end of list, call loadMore if hasMore

    3. Create `signal_detail_screen.dart`:

       - Signal data section: formatted JSON representation using JsonEncoder.withIndent
       - Copy-to-clipboard button for signal data
       - Back navigation to inbox
