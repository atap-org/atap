# Phase 3: Mobile App - Context

**Gathered:** 2026-03-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Flutter mobile app with human entity onboarding via claim links, inbox view with real-time signal display, and push notifications. Includes platform-side work: human entity registration, claim/invite flow, email attestation for Trust Level 1, delegation creation, push token storage, and FCM/APNs delivery. This phase expands beyond the original agent-only mobile scope to deliver the full human onboarding experience.

**Platform additions required:**
- Human entity registration endpoint (key derived from Ed25519 pubkey)
- Claim link generation (`POST /v1/claims`) and redemption
- Email attestation and verification
- Delegation document creation (human → agent)
- Push token storage per entity and push delivery on new signal

**Mobile app:**
- Claim link deep linking and fallback landing page
- Human onboarding flow (claim → email verify → registration)
- Inbox view with SSE streaming
- Push notification handling (FCM + APNs)

</domain>

<decisions>
## Implementation Decisions

### Key storage & authentication
- Ed25519 private key stored in OS secure enclave (iOS Keychain / Android Keystore)
- Biometric authentication (Face ID / fingerprint) required for every signed API action
- Key loss handling deferred to Phase 2 (human key recovery with encrypted server-side storage)
- For Phase 3: lost key means re-register as new identity (same as agents)
- Ed25519 compatibility between Flutter (Dart library) and Go stdlib must be validated as the first task — generate keypair in Flutter, sign payload, verify against Go platform

### Inbox presentation
- Card-based list layout: each signal as a card showing sender, type, timestamp, data preview, priority indicator
- Signal detail view: structured fields for mandatory parts (route, trust, context) plus formatted JSON representation for signal.data payload
- Real-time updates via SSE live stream while app is in foreground, plus pull-to-refresh as manual fallback
- Empty inbox: minimal placeholder ("No signals yet" with pull-to-refresh hint)

### Push notifications
- All signals trigger a push notification (no filtering by priority in Phase 3)
- Notification content: signal type + sender (e.g., "task.completed from agent://abc123") — no payload preview for privacy
- Push delivery logic lives platform-side: Go FCM/APNs client integrated into the platform binary
- Platform stores push token per entity, sends push on every new signal
- Both iOS (APNs) and Android (FCM) supported from day one

### Registration & onboarding flow
- Humans are onboarded through invite links generated by agents
- Agent creates claim via `POST /v1/claims`, returns link: `https://link.atap.app/claim/{claim-code}`
- Claim code format: `ATAP-` + 4 alphanumeric chars (per existing ID conventions)
- Deep link behavior: if app installed → deep link into app onboarding flow; if not installed → landing page at atap.app with app store links and claim code displayed as text
- Email verification required during onboarding — human starts at Trust Level 1
- Phone, World ID (Level 2), eID (Level 3) are future trust level upgrades, not in Phase 3
- After claiming: a delegation document (`del_`) is created linking human → agent (human has authority over the inviting agent)
- Human ID derived from Ed25519 public key: `lowercase(base32(sha256(pubkey))[:16])` (per spec)

### Claude's Discretion
- Flutter project structure and state management approach
- Dart Ed25519 library selection (based on compat validation results)
- Deep link implementation (uni_links vs go_router deep linking vs app_links)
- FCM/APNs Go client library selection
- Push token registration endpoint design
- Claim link landing page design and hosting
- Migration schema for human entities, claims, delegations, push tokens
- Email verification implementation (send mechanism, verification flow)

</decisions>

<specifics>
## Specific Ideas

- Claim links follow pattern: `https://link.atap.app/claim/ATAP-X7K2`
- Landing page fallback is simple — app store buttons + "Enter this code after installing: ATAP-X7K2"
- The claim flow establishes the core ATAP trust chain: human delegates authority to agent
- Biometrics on every action — this is a trust protocol app, security is the product

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `internal/crypto/crypto.go`: Ed25519 key generation, canonical JSON (JCS), signature verification — reuse for human key generation and claim signing
- `internal/models/models.go`: Entity model with type field supporting "human" — extend with Claim, Delegation, PushToken models
- `internal/store/store.go`: pgx/v5 pool pattern with interface-based store — extend with ClaimStore, DelegationStore, PushTokenStore
- `internal/api/api.go`: Handler struct, auth middleware, RFC 7807 errors — extend with claim, delegation, push endpoints
- Existing ID conventions: `clm_` + 12 hex for claims, `del_` + 12 hex for delegations, `ATAP-` + 4 alphanum for claim codes

### Established Patterns
- EntityStore interface pattern for testability — replicate for new store interfaces
- Auth middleware sets `c.Locals("entity")` — all new endpoints use this
- Ed25519 signed request auth (not bearer tokens)
- Fiber v2 route groups with middleware chaining
- Redis pub/sub for real-time delivery — extend for push notification triggering

### Integration Points
- `api.SetupRoutes()` — add claim, delegation, push token endpoints
- `cmd/server/main.go` — add FCM/APNs client initialization
- `migrations/` — add migrations for human entities, claims, delegations, push tokens
- Redis pub/sub subscriber — add push notification delivery alongside SSE

</code_context>
