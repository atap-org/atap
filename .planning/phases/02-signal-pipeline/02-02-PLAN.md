**4. Implement SendSignal handler** (POST /v1/inbox/:entityId):
- Get authenticated entity from `c.Locals("entity")`
- Get targetID from params
- Check Content-Length and body size <= MaxSignalPayload (64KB), return 413 if exceeded
- Parse request body into a signal send request (route + signal + context fields)
- Validate target entity exists via entityStore.GetEntity
- Build models.Signal: generate ID with crypto.NewSignalID(), set Version "1", TS to now, populate Route (origin = sender URI, target = target URI), set Trust block (level = sender trust level, signer = sender URI, signer_key_id = sender key ID)
- Sender must sign the signal: use crypto.SignablePayload(route, signal) then crypto.Sign(senderPrivateKey, payload) -- BUT the sender provides the signature in the request. Verify it using crypto.Verify(senderPublicKey, signablePayload, signature). Store the verified signature.
- Actually, per CONTEXT.md: "Every signal is Ed25519 signed by the sender". The CLIENT sends the signature. The server receives `trust.signature` and `trust.signer_key_id` in the request body, verifies the signature matches the authenticated entity's public key, and stores it.
- Set Context.Source = "agent", compute ExpiresAt from TS + TTL (or DefaultSignalTTL), set DeliveryStatus = "pending"
- Call signalStore.SaveSignal -- if ErrDuplicateSignal, return 409 Conflict
- After successful save, publish to Redis: `h.redis.Publish(ctx, "inbox:"+targetID, signalJSON)` -- this is the write-then-notify pattern (SSE-04)
- Return 202 Accepted with the signal (including ID and timestamps)

**5. Implement GetInbox handler** (GET /v1/inbox/:entityId):
- Auth required. Verify authenticated entity ID matches :entityId (entities can only read their own inbox)
- Parse query params: `after` (cursor string), `limit` (int, default 50, max 100)
- Call signalStore.GetInbox(ctx, entityID, after, limit)
- Return InboxResponse with signals, hasMore, cursor (= last signal ID if hasMore)

**6. Implement InboxStream handler** (GET /v1/inbox/:entityId/stream):
- Auth required. Verify authenticated entity ID matches :entityId
- Read Last-Event-ID header for SSE reconnection replay
- Set SSE headers: Content-Type text/event-stream, Cache-Control no-cache, Connection keep-alive, X-Accel-Buffering no
- Use c.Context().SetBodyStreamWriter(fasthttp.StreamWriter(func(w *bufio.Writer) { ... })):
  a. Subscribe to Redis FIRST: `sub := h.redis.Subscribe(ctx, "inbox:"+entityID)`, defer sub.Close(). Get channel: `ch := sub.Channel()`
  b. Replay missed signals from PostgreSQL if Last-Event-ID is set: call signalStore.GetSignalsAfter(ctx, entityID, lastEventID), write each as SSE event
  c. Start heartbeat ticker at 30 seconds
  d. Select loop: Redis message -> write SSE event, ticker -> write heartbeat comment (": heartbeat\n\n"), check w.Flush() error for disconnect detection
- SSE event format: `event: signal\nid: {signalID}\ndata: {signalJSON}\n\n`
- Subscribe first, then replay, to avoid the replay gap (per RESEARCH.md Pitfall 1). Clients must handle potential duplicates by signal ID.
- Import "bufio" and "github.com/valyala/fasthttp" for StreamWriter
